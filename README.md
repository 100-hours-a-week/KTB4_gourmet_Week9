## 1. 8주차 피드백 반영

8주차 과제를 제출한 뒤 인증·인가와 테스트 코드에 대한 피드백을 받았다. 처음에는 JWT 발급과 HttpOnly Cookie 적용에 집중하면서 인증 구조를 어느 정도 완성했다고 생각했지만, 비밀번호 평문 저장과 리소스 소유자 검증 누락 등 기본적인 보안 문제를 놓치고 있었다.

가장 먼저 `BCryptPasswordEncoder`를 Bean으로 등록하고, 회원가입과 비밀번호 변경 시 `encode()`를 사용해 비밀번호를 암호화하도록 수정했다. 로그인에서는 기존의 문자열 `equals()` 비교를 제거하고, 사용자가 입력한 비밀번호와 DB에 저장된 암호화 비밀번호를 `matches()`로 비교하도록 변경했다.

```java
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder();
}
```

처음에는 인증이 필요한 API에 `.authenticated()`를 적용하면 인가까지 충분히 처리된다고 생각했다. 하지만 Postman으로 로그인한 다른 사용자가 게시글 삭제 API를 직접 호출했을 때 `204 No Content`로 삭제가 처리되는 문제를 확인했다.

`.authenticated()`는 사용자의 로그인 여부만 확인할 뿐, 현재 로그인한 사용자가 해당 리소스의 작성자인지는 판단하지 않았다. 이를 해결하기 위해 `SecurityContext`에서 로그인 사용자 id를 가져오는 `SecurityUtil`을 만들고, 로그인 사용자와 요청 대상 사용자를 비교하도록 수정했다.

```java
public static void validateLoginUser(Long targetUserId) {
    Long loginUserId = getLoginUserId();

    if (!loginUserId.equals(targetUserId)) {
        throw new ForbiddenException("접근 권한이 없습니다.");
    }
}
```

해당 검증은 다음 기능에 적용했다.

- 회원정보 수정
- 비밀번호 변경
- 회원 탈퇴
- 게시글 작성·수정·삭제
- 댓글 작성·수정·삭제
- 좋아요 상태 조회 및 등록·취소

권한이 없는 요청은 `ForbiddenException`을 발생시키고, `GlobalExceptionHandler`에서 `403 Forbidden`으로 처리하도록 구성했다.

이 과정을 통해 401과 403의 차이도 명확하게 이해할 수 있었다.

- `401 Unauthorized`: 로그인하지 않았거나 토큰이 없거나 유효하지 않은 경우
- `403 Forbidden`: 로그인은 되어 있지만 해당 리소스를 조작할 권한이 없는 경우

프론트에서 수정·삭제 버튼을 숨기는 것은 사용자 경험을 위한 처리일 뿐이며, 실제 보안은 백엔드에서 직접 요청자를 검증해야 한다는 점을 알게 되었다.

## 2. 테스트 코드 개선

기존 `UserDomainTests`는 회원 생성 후 필드 값을 다시 읽거나, 닉네임과 비밀번호가 전달받은 값으로 변경되는지만 확인하는 테스트가 대부분이었다.

테스트는 통과했지만 실제 서비스에서 발생할 수 있는 중복 가입, 로그인 실패, 비밀번호 평문 저장, 권한 없는 요청 등의 문제는 검증하지 못했다.

피드백을 반영해 Mockito 기반의 `UserServiceTest`를 작성하고 다음 내용을 검증했다.

- 회원가입 시 비밀번호가 BCrypt 해시로 저장되는지
- 비밀번호 변경 시에도 암호화가 적용되는지
- 중복 이메일 회원가입이 거절되는지
- 중복 닉네임 회원가입이 거절되는지
- 잘못된 비밀번호로 로그인이 거절되는지
- 탈퇴한 사용자의 로그인이 거절되는지
- 다른 사용자의 비밀번호 변경 요청이 권한 예외로 처리되는지

회원 탈퇴 테스트도 수정했다. 기존 테스트는 저장되지 않은 User 객체를 사용했기 때문에 id가 `null`인 상태에서도 다음과 같은 값으로 테스트가 통과할 수 있었다.

```text
deleted_user_null@deleted.local
```

이를 해결하기 위해 `DataJpaTest`로 회원을 실제 H2 DB에 저장하고, 생성된 id를 이용해 탈퇴 후 이메일과 닉네임이 정확하게 변경되는지 검증했다.

```text
deleted_user_{id}@deleted.local
탈퇴회원_{id}
```

최종적으로 실행되는 테스트 결과는 다음과 같다.

```text
UserServiceTest: 7개 통과
UserDeleteJpaTest: 1개 통과
```

기본으로 생성된 `AssignmentApplicationTests`의 `contextLoads()`는 Spring Context 실행 여부만 확인하고 실제 기능을 검증하지 못해 삭제했다.

`MockMvc`와 `WebMvcTest`를 이용해 401, 403, 409 응답을 자동화하는 테스트도 시도했지만 테스트 Context 구성 과정에서 문제를 해결하지 못했다. 해당 테스트는 삭제하지 않고 `@Disabled`로 남겨두었으며, 현재는 Postman을 통해 인증 실패와 권한 실패 응답을 직접 확인했다.

앞으로는 단순히 테스트가 통과하는지를 보는 것이 아니라, 실제 버그가 발생했을 때 테스트가 실패할 수 있는지를 기준으로 테스트를 작성해야겠다고 생각했다.

## 3. 프론트엔드 디자인 변경

이번 과제에서는 기존 아무말 대잔치 결과물과 차별화하기 위해 Gourmet Community만의 디자인을 새롭게 구성했다.

먼저 서비스의 브랜드를 표현하기 위해 다크 그린 계열의 전용 로고를 제작했다. 거꾸로 된 오각형 형태 안에 대문자 `G`를 배치하고, 아래쪽에 `Gourmet Community` 문구를 넣어 커뮤니티의 정체성을 표현했다.

로그인 화면은 기존의 단순한 회색 배경과 입력 폼 형태에서 벗어나 다음과 같이 변경했다.

- 다크 그린과 크림색 중심의 색상 구성
- Gourmet 전용 로고 적용
- 브랜드 소개 영역과 로그인 카드 영역 분리
- 입력창과 버튼 디자인 변경
- 오류 메시지 스타일 변경
- 비밀번호 보기·숨기기 기능 추가
- 모바일 환경을 고려한 반응형 레이아웃 적용
- 기존 HttpOnly Cookie 로그인 흐름 유지

디자인을 변경하면서 기존 JavaScript가 참조하는 `id`와 API 요청 구조는 최대한 유지했다. 화면을 변경하는 과정에서 기존 로그인 기능이 깨지지 않도록 기능과 디자인의 역할을 분리하려고 했다.

현재는 로그인 화면을 먼저 변경했으며, 이후 회원가입 화면과 게시글 목록, 상세 페이지도 같은 다크 그린 브랜드 스타일로 통일할 예정이다.

## 4. 커뮤니티 구조 변경에 대한 고민

기존 서비스는 모든 게시글이 하나의 목록에 표시되는 형태였다. 디자인을 변경하면서 단순히 색상과 레이아웃만 바꾸기보다, 서비스 자체를 여러 목적의 게시판으로 확장하는 방향을 고민했다.

예정한 게시판은 다음과 같다.

- 자유게시판
- 질문게시판
- 학습 기록 일지
- 프로젝트 모집

화면 왼쪽에는 열고 닫을 수 있는 내비게이션을 배치하고, 하나의 게시판 화면을 재사용하면서 선택한 게시판에 따라 제목과 게시글 목록 데이터만 변경하려 한다.

게시판마다 HTML 파일을 복제하는 방식이 아니라, 공통 화면에서 `boardType`을 기준으로 데이터를 조회하는 방식을 사용할 예정이다.

## 5. 추가 기능에 대한 고민

추가 기능을 선정할 때는 단순히 화면에 새로운 버튼을 추가하는 것보다, 데이터 정합성이나 동시성, 검색 성능, 캐시 등 백엔드에서 설명할 수 있는 기술적 깊이가 있는지를 중요하게 생각했다.

### 5.1 게시판 분류 DB 정식화

`Post`에 `BoardType`을 추가하고 게시판 유형을 DB에서 관리하려 한다.

```java
public enum BoardType {
    FREE,
    QUESTION,
    STUDY,
    PROJECT
}
```

게시글 작성 시 게시판 유형을 함께 저장하고, 게시글 목록 조회 시 게시판 유형을 조건으로 사용한다.

```http
GET /posts?boardType=QUESTION&page=0&size=10
```

이 과정에서 다음 내용을 고민하려 한다.

- Enum을 문자열로 저장할지 숫자로 저장할지
- 잘못된 게시판 유형의 예외 처리
- 게시판 유형별 목록 조회 쿼리
- `board_type`, `created_at`, `post_id` 복합 인덱스
- localStorage를 이용한 임시 게시판 구분 제거
- 게시판이 동적으로 추가될 경우 Enum 방식이 가지는 한계

이번 프로젝트에서는 게시판 종류가 고정되어 있으므로 우선 Enum으로 구현할 예정이다.

---

### 5.2 게시글 검색

게시판이 여러 개로 나뉘면 사용자가 원하는 글을 찾기 어려울 수 있기 때문에 제목과 내용을 기준으로 검색하는 기능을 추가하려 한다.

```http
GET /posts/search?keyword=spring&boardType=QUESTION
```

초기에는 `LIKE` 검색을 사용할 수 있다.

```sql
WHERE title LIKE '%spring%'
   OR content LIKE '%spring%'
```

하지만 검색어 앞에 `%`가 붙으면 일반 인덱스를 활용하기 어렵기 때문에 데이터가 증가했을 때 다음 방식을 검토하려 한다.

- Full Text Index
- 제목과 내용 검색 조건 분리
- 게시판 유형과 검색어를 조합한 조회
- 검색 결과 페이지네이션
- 검색 전용 서버 도입 가능성

검색창만 추가하는 데서 끝내지 않고, 검색 쿼리와 인덱스의 관계까지 확인하는 방향으로 구현하려 한다.

### 5.3 인기글

메인 화면이나 게시판 사이드 영역에 최근 인기 게시글을 보여주는 기능을 추가하려 한다.

누적 조회수만 기준으로 하면 오래된 게시글이 계속 상단에 남을 수 있기 때문에, 최근 일정 기간의 좋아요, 댓글, 조회수를 조합해 인기 점수를 계산하려 한다.

예시 점수는 다음과 같다.

```text
인기 점수
= 좋아요 수 × 5
+ 댓글 수 × 3
+ 조회수 × 0.1
```

예상 조건은 다음과 같다.

- 최근 7일 이내 작성된 게시글
- 삭제된 게시글 제외
- 최소 조회수 또는 최소 반응 수 적용
- 같은 점수라면 최신 게시글 우선
- 전체 인기글 또는 게시판별 인기글 제공

초기에는 DB 집계 쿼리로 구현하고, 조회 요청이 많아지면 Redis Sorted Set이나 별도의 집계 테이블 사용을 검토하려 한다.

### 5.4 주간 활동 랭킹

게시글 작성, 댓글 작성, 좋아요 획득, 답변 채택 등에 포인트를 지급하고 최근 7일간의 활동을 기준으로 사용자 랭킹을 제공하는 기능을 고민했다.

예시 점수는 다음과 같다.

```text
게시글 작성                       +10점
댓글 작성                          +3점
내 게시글이 좋아요를 받은 경우     +2점
질문 게시판에서 답변이 채택된 경우 +15점
```

단순히 글이나 댓글을 많이 작성한 사용자에게 높은 점수를 주면 의미 없는 콘텐츠를 반복 작성할 가능성이 있다. 이를 방지하기 위해 다음과 같은 제한 조건이 필요하다고 생각했다.

- 본인이 본인 게시글에 누른 좋아요는 점수에서 제외
- 동일 사용자의 같은 게시글 좋아요는 한 번만 인정
- 좋아요 취소 시 점수 회수
- 삭제된 게시글과 댓글의 점수 처리
- 댓글 작성 점수는 하루 최대 횟수 제한
- 게시글 작성 점수도 하루 최대 횟수 제한
- 탈퇴 회원은 랭킹에서 제외
- 최근 7일간 발생한 활동만 반영

포인트 지급 이력을 추적하기 위해 다음과 같은 별도 테이블을 고려하고 있다.

```text
point_histories

- point_history_id
- user_id
- activity_type
- point
- reference_type
- reference_id
- created_at
```

초기에는 최근 7일간의 포인트 이력을 DB에서 집계하고, 요청량이 많아지면 Redis Sorted Set을 이용하는 방향을 검토하려 한다.

### 5.5 실시간 알림

내 게시글에 댓글이나 좋아요가 추가되었을 때 알림 아이콘에 숫자를 표시하고, 알림 목록에서 내용을 확인할 수 있는 기능을 추가하려 한다.

알림 종류는 우선 다음 두 가지로 제한한다.

- 내 게시글에 댓글이 작성된 경우
- 내 게시글에 좋아요가 추가된 경우

채팅처럼 클라이언트와 서버가 계속 양방향으로 메시지를 교환하는 기능은 아니기 때문에 WebSocket보다 서버에서 클라이언트로 이벤트를 전달하는 SSE가 더 적절할 수 있다고 생각했다.

예상 흐름은 다음과 같다.

```text
댓글 또는 좋아요 생성
→ 애플리케이션 이벤트 발행
→ 알림 DB 저장
→ 접속 중인 사용자에게 SSE 전송
→ 알림 뱃지 증가
```

실시간 전송만 구현하면 사용자가 접속하지 않았을 때 발생한 알림을 확인할 수 없기 때문에, 알림은 DB에 저장하고 SSE는 화면에 빠르게 반영하기 위한 수단으로 사용하려 한다.

### 5.6 좋아요 동시성 처리

현재 좋아요 테이블에는 한 사용자가 같은 게시글에 중복으로 좋아요를 누르지 못하도록 다음 제약이 존재한다.

```text
UNIQUE(user_id, post_id)
```

하지만 동시에 같은 요청이 여러 번 들어오면 다음과 같은 경쟁 상태가 발생할 수 있다.

```text
요청 A: 좋아요 존재 여부 확인 → 없음
요청 B: 좋아요 존재 여부 확인 → 없음
요청 A: 좋아요 저장
요청 B: 좋아요 저장 시도
```

DB의 유니크 제약으로 중복 데이터는 방지할 수 있지만, 두 번째 요청에서는 예외가 발생할 수 있다.

또한 현재 좋아요 API는 한 번 호출할 때마다 등록과 취소가 바뀌는 토글 방식이다. 요청이 재전송되면 사용자가 의도한 상태와 반대로 변경될 수 있기 때문에 다음과 같이 API를 분리하는 방향도 고려하고 있다.

```http
PUT /posts/{postId}/likes
DELETE /posts/{postId}/likes
```

- `PUT`: 이미 좋아요 상태여도 최종 결과는 좋아요 상태
- `DELETE`: 이미 취소 상태여도 최종 결과는 취소 상태

이를 통해 좋아요 API를 멱등하게 만들고, 동시 요청과 재시도 상황에서 데이터 정합성을 높이려 한다.

---

### 5.7 조회수 동시성 처리

현재 조회수는 로그인 사용자가 같은 게시글을 처음 조회했을 때만 증가하도록 `PostView`를 저장하고 있다.

하지만 여러 사용자가 같은 게시글을 동시에 조회하면 다음과 같은 갱신 손실이 발생할 수 있다.

```text
현재 조회수: 100

요청 A가 100을 읽고 101로 변경
요청 B도 100을 읽고 101로 변경

최종 조회수: 101
기대 조회수: 102
```

이를 해결하기 위해 엔티티 값을 읽은 뒤 변경하는 방식보다 DB에서 직접 조회수를 증가시키는 원자적 쿼리를 사용할 수 있다.

```sql
UPDATE posts
SET view_count = view_count + 1
WHERE post_id = ?
```

또한 다음과 같은 방법도 비교해볼 수 있다.

- 낙관적 락과 재시도
- 비관적 락
- 원자적 UPDATE 쿼리
- Redis에 조회수를 누적한 뒤 일정 주기로 DB 반영
- `PostView` 중복 저장 방지를 위한 유니크 제약

조회수는 결제나 재고처럼 반드시 완벽하게 일치해야 하는 데이터인지도 함께 판단해, 정확성과 성능 사이의 기준을 정해야 한다고 생각했다.

## 6. 전체 회고

이번 피드백을 반영하면서 기능이 정상적으로 보이는 것과 실제로 안전하게 동작하는 것은 다르다는 점을 느꼈다.

로그인한 사용자에게 수정·삭제 버튼이 보이지 않는다고 해서 인가 처리가 끝난 것이 아니었고, 테스트가 통과한다고 해서 실제 버그를 잡을 수 있는 테스트라는 의미도 아니었다.

또한 프론트엔드 디자인 변경도 단순히 색상과 폰트를 바꾸는 작업이 아니라, 서비스의 콘셉트와 이후 추가할 기능을 함께 고려해야 하는 작업이었다. Gourmet Community라는 브랜드와 다중 게시판 구조를 먼저 정하고 나니 필요한 화면과 기능도 더 구체적으로 생각할 수 있었다.

이번주 과제를 빨리 진행하지 못했다. 디자인 개선이 메인이라 생각을 해서 그런지 어떻게 디자인을 변경해야 할지, 어떤 기능을 추가해야 할지 고민을 많이 했던거 같다. 고민을 하면서 미루다 보니까 과제 제출 시기를 놓쳤고 우선 뭐라도 빨리 제출하는 게 낫다고 판단하여 회고와 디테일이 부족하지 않을까 라는 걱정도 되고 AI의 도움을 더 많이 받았다고 느껴지는 주차다. 

이제 점차 사람마다 구현하는 기능이 다를텐데 좀 더 AI를 쓰는것도 중요하다 생각하지만 시간 투자를 많이 하여 얻어가는 것이 더 많았으면 생각한다. 과제에 대한 몰입도를 생각해보면 아쉬운 주차라고 생각을 한다.
